@Entity                              // Maps to database table
@Table(name = "claims")              // Explicit table name
public class Claim {
    
    @Id                                  // Primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long claimId;
    
    @Column(unique = true, nullable = false)
    private String claimNumber;
    
    @ManyToOne(fetch = FetchType.LAZY)   // Foreign key relationship
    @JoinColumn(name = "policy_id")
    private Policy policy;
    
    @Enumerated(EnumType.STRING)         // Store enum as text
    private ClaimStatus claimStatus;
    
    @PrePersist                          // Before INSERT callback
    protected void onCreate() {
        createdDate = LocalDateTime.now();
    }
}

@Configuration
@EnableWebSecurity
@EnableMethodSecurity      // Enables @PreAuthorize
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) {
        http
            .authorizeHttpRequests(auth -> auth
                // Public pages
                .requestMatchers("/login", "/register").permitAll()
                
                // Role-based access
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers("/claims/**").authenticated()
            )
            .formLogin(form -> form
                .loginPage("/login")
                .defaultSuccessUrl("/dashboard")
            );
        return http.build();
    }
}

When submitting a claim:
  • User must be enrolled in the policy (ACTIVE status)
  • Claim amount cannot exceed policy coverage amount
  • Policy and user must exist in database
public Claim submitClaim(ClaimDTO dto, User claimant) {
    
    // 1. Validate policy exists
    Policy policy = policyRepository.findById(dto.getPolicyId())
        .orElseThrow(() -> new RuntimeException("Policy not found"));
    
    // 2. Validate enrollment
    boolean isEnrolled = enrollmentRepository
        .existsByPolicyholderAndPolicyAndStatus(claimant, policy, ACTIVE);
    
    if (!isEnrolled) {
        throw new RuntimeException("User not enrolled");
    }
    
    // 3. Validate amount
    if (dto.getClaimAmount().compareTo(policy.getCoverageAmount()) > 0) {
        throw new RuntimeException("Amount exceeds coverage");
    }
    
    // 4. Create and save
    Claim claim = Claim.builder()
        .claimNumber(generateClaimNumber())  // CLM-20260201-001
        .policy(policy)
        .claimant(claimant)
        .claimAmount(dto.getClaimAmount())
        .claimStatus(ClaimStatus.PENDING)
        .build();
    
    return claimRepository.save(claim);
}

üîí Ownership Validation

@PostMapping("/cancel/{id}")
@PreAuthorize("hasRole('USER')")
public String cancelClaim(@PathVariable Long id,
                          @AuthenticationPrincipal UserDetails userDetails) {
    
    User user = userService.findByUsername(userDetails.getUsername());
    Claim claim = claimService.findById(id).orElseThrow();
    
    // ‚ö†Ô∏è CRITICAL: Verify ownership
    if (!claim.getClaimant().getUserId().equals(user.getUserId())) {
        throw new RuntimeException("Cannot cancel others' claims!");
    }
    
    // Only PENDING claims can be cancelled
    if (claim.getClaimStatus() != ClaimStatus.PENDING) {
        throw new RuntimeException("Only pending claims cancellable");
    }
    
    claimService.updateStatus(id, ClaimStatus.CANCELLED);
}

6. Key Definitions

Term Definition
üîë Authentication "Who are you?"
The process of verifying identity using credentials (username/password). In HICMS: Form login ‚Üí BCrypt password verification ‚Üí Session creation.
üõ°Ô∏è Authorization "What can you do?"
Determining if authenticated user can access a resource. In HICMS: URL patterns in SecurityConfig + @PreAuthorize on methods.
üé´ JWT JSON Web Token
Compact token format: header.payload.signature
Note: HICMS uses session-based auth, not JWT.
üìã Claims (Security) Pieces of information about a subject (user) in authentication context. Examples: username, roles, expiration time. In HICMS: Stored in UserDetails within SecurityContext.

7. Interview Questions & Answers

üî∑ Spring Boot Annotations

What does @SpringBootApplication do internally?
It's a convenience annotation combining three annotations:
  • @Configuration - Marks class as source of bean definitions
  • @EnableAutoConfiguration - Enables Spring Boot's auto-config
  • @ComponentScan - Scans current package and sub-packages
Difference between @Controller and @RestController?
@Controller @RestController
Returns view names (Thymeleaf) Returns data directly (JSON)
Need @ResponseBody for JSON @ResponseBody included
How does @Transactional work?
Spring creates a proxy around the bean:
  1. Before method: Start transaction
  2. Method executes normally
  3. On success: Commit transaction
  4. On exception: Rollback transaction
@Transactional(readOnly = true) optimizes for read operations.
Explain FetchType.LAZY vs EAGER
LAZY (Recommended) EAGER
Load when accessed Load immediately with parent
Separate SQL query JOIN in same query
Better performance Can cause N+1 if not careful

üî∑ Security Questions

How does Spring Security authentication work?
  1. User submits credentials (POST /login)
  2. UsernamePasswordAuthenticationFilter intercepts
  3. AuthenticationManager delegates to DaoAuthenticationProvider
  4. CustomUserDetailsService.loadUserByUsername() queries DB
  5. BCryptPasswordEncoder verifies password
  6. On success: Create Authentication ‚Üí Store in SecurityContext ‚Üí Create Session
What is SecurityContextHolder?
It's the core storage for security information: 
SecurityContextHolder
  ‚îî‚îÄ‚îÄ SecurityContext
        ‚îî‚îÄ‚îÄ Authentication
              ‚îú‚îÄ‚îÄ Principal (UserDetails)
              ‚îú‚îÄ‚îÄ Credentials (password)
              ‚îú‚îÄ‚îÄ Authorities (roles)
              ‚îî‚îÄ‚îÄ isAuthenticated (boolean)
Uses ThreadLocal by default - each thread has its own context.
Explain @PreAuthorize and how it differs from URL security
URL Security (SecurityConfig) @PreAuthorize (Method)
Coarse-grained (URL patterns) Fine-grained (per method)
Evaluated before controller Evaluated before method
Cannot access method params Can use SpEL with params
@PreAuthorize("hasRole('ADMIN') and #id == principal.id")

üî∑ Claim Module Questions

Explain the claim lifecycle in HICMS

PENDING ‚Üí Initial state after submission

UNDER_REVIEW ‚Üí Adjuster assigns self

APPROVED ‚Üí Adjuster approves with amount

REJECTED ‚Üí Adjuster rejects with remarks

CANCELLED ‚Üí User cancels (only from PENDING)

What validations occur during claim submission?
  1. Policy exists - Verify policy ID is valid
  2. User enrolled - Check active enrollment in policy
  3. Amount valid - Cannot exceed coverage amount
  4. DTO validation - @NotNull, @DecimalMin, @NotBlank
Why is ownership validation important?
Without ownership checks, users could:
  • Edit other users' claims
  • Cancel other users' claims
  • Access sensitive claim data
Always verify: claim.getClaimant().getUserId().equals(currentUser.getUserId())

üî∑ Advanced Questions

What is the N+1 query problem?
Loading N entities, then lazily loading each relationship = 1 + N queries. 
List<Claim> claims = claimRepository.findAll();  // 1 query
for (Claim c : claims) {
    c.getPolicy().getName();  // N queries!
}
Solutions:
  • JOIN FETCH in JPQL
  • @EntityGraph
  • DTO projections
Why @Enumerated(STRING) over ORDINAL?
STRING (Recommended) ORDINAL
Stores "PENDING" Stores 0
Readable in database Numbers are confusing
Safe to reorder enum Reorder breaks data!
How does @Valid work with BindingResult?
@Valid triggers Bean Validation (@NotNull, @Size, etc.).
BindingResult must immediately follow the @Valid parameter: 
public String submit(
    @Valid @ModelAttribute ClaimDTO dto,  // ‚úì
    BindingResult result,                  // Must be right after!
    Model model) {
    
    if (result.hasErrors()) {
        return "form";  // Show errors
    }
}
